# Capacitor Android: compile & distribute

Everything runs from the web project root. If the project has Makefile lanes
(e.g. `make android-sync` / `apk-debug`), prefer them; the raw commands below
are what such lanes wrap.

## Ground rules

1. **Treat `android/` as git-ignored and disposable** — regenerated by
   `npx cap add android`. Never park config there. Durable config lives in:
   - An idempotent **config script** (pattern: `scripts/android-permissions.sh`,
     run after every `cap sync`) for manifest edits: permissions the plugins
     need (e.g. CAMERA), stripping unwanted ones (e.g. RECORD_AUDIO so
     "Microphone" stays off the store listing), orientation lock on
     MainActivity. Add future manifest edits there.
   - The Makefile / env — exported `ANDROID_HOME`/`JAVA_HOME` pins, plus any
     copy steps for git-ignored secrets (e.g. `google-services.json` for FCM).
   - Icon/splash source image + a `capacitor-assets` step per sync.
   - `capacitor.config.ts` — the `appId`'s single source of truth.
2. **Toolchain pins (Capacitor 8):** JDK **21** (17 fails with
   `invalid source release: 21`), SDK platform **android-36**
   (`compileSdkVersion` in the generated `variables.gradle`). Override per
   machine on the command line, not in `android/local.properties` (dies on
   regen): `JAVA_HOME=/path/jdk21 ANDROID_HOME=~/Android/Sdk`.
3. **Changing the appId requires regenerating `android/`**
   (`rm -rf android && npx cap add android`) — `cap sync` does NOT rewrite
   `applicationId`. A renamed app installs **alongside** the old one (fresh
   local data), and Google Cloud Android OAuth clients + Firebase Android
   apps are keyed to the package name — they must be re-created for it.

## Compile

```bash
npm run build && npx cap sync android    # + config script + assets + secret copies
cd android && ./gradlew assembleDebug    # → app/build/outputs/apk/debug/app-debug.apk
```

- The **debug APK is signed with the auto-generated debug key and installs
  directly** — it's the device-testing lane. `assembleRelease` without a
  signing config emits an *unsigned* APK that Android refuses to install.
- After regenerating `android/`, the first build re-downloads Gradle — slow
  once, then cached.
- **Sanity-check the bundle before shipping it anywhere**: grep the synced
  web assets for a dev/localhost API URL
  (`grep -rl "localhost" android/app/src/main/assets/public/`) — a leftover
  env-specific build poisons the assets; re-run the sync to rebuild.

## Distribute — ad-hoc device testing

Any file-share channel works (Drive/Dropbox/USB). Conventions that keep it
sane:

- Name builds `<app>-debug-<git short hash>.apk` so a device build maps to a
  commit.
- Keep exactly ONE current APK at the share location's top level; move older
  ones into an `old_builds/` subfolder *before* copying the new one, so the
  phone's file picker always shows just the latest.

## Distribute — signed release / Play Store

Build it in this order:

1. **Keystore**: generate once
   (`keytool -genkeypair -v -keystore upload.jks -keyalg RSA -keysize 2048
   -validity 10000 -alias upload`), store it OUTSIDE the repo and out of
   git; back it up. Prefer **Play App Signing** with this as the upload key
   (the default and recommended) — then a lost upload key is recoverable,
   a lost app-signing key is not.
2. **Wire signing without touching `android/` by hand**: keep
   `keystore.properties` next to the keystore; extend the config script to
   inject `signingConfigs` into the generated `android/app/build.gradle`
   idempotently on every sync. Never hand-edit the generated gradle file.
3. **Version bump per upload**: Play requires a strictly increasing
   `versionCode`; the generated `build.gradle` hardcodes `versionCode 1`, so
   the same inject-script must stamp it — `git rev-list --count HEAD` is a
   free monotonic source. `versionName` is the user-facing version.
4. **Play uploads want an AAB**, not an APK:
   `cd android && ./gradlew bundleRelease` →
   `app/build/outputs/bundle/release/app-release.aab`.
5. **Play Console** (one-time $25): create the app, complete Data safety
   (mirror the iOS privacy declarations if both stores are targeted),
   content rating, then release via the **Internal testing** track first
   (the Play analog of TestFlight — instant, up to 100 testers via link)
   before production.
6. **Google sign-in in release builds**: the Android OAuth client in Google
   Cloud needs the **upload key's SHA-1** (`keytool -list -v -keystore …`),
   and once Play App Signing takes over, **Play's app-signing SHA-1 too**
   (Play Console → App integrity). Missing SHA-1s fail silently at runtime.

## Known gotchas

- The OS camera/photo-picker is a separate Android activity and the OS may
  kill the app behind it (`appRestoredResult` recovery is a web-layer
  concern) — don't try to "fix" missing-photo bugs in native config.
- Push (FCM) needs the git-ignored `google-services.json` copied into
  `android/app/` on every sync; without it the APK builds fine but push
  registration fails silently.
- Debug and release builds have different signatures — a device can hold
  one or the other, not both signature-variants of the same appId; expect
  an uninstall when switching lanes.
